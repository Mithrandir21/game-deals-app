- [0] Compose `Row.verticalAlignment = CenterVertically` centers within the inner content area, not the padded outer bounds — for inter-row spacing on a centered row, use `padding(vertical = X)` not `padding(bottom = X)`, or the row's content looks pushed toward the top.
- [0] Compose `onNodeWithText` is exact-match by default — when a single `Text` uses `buildAnnotatedString` to concatenate spans, the whole concatenated string is one node; assertions must use the full string or pass `substring = true`.
- [0] When a Composable gets a new required parameter (e.g. `onViewFavourites`), every test invocation of that Composable must add the param OR a default value must be supplied — Compose instrumented tests fail at compile, not at test discovery, so a stale test file silently rots a whole module's Android tests.
- [0] When a ViewModel adds a new collected StateFlow property, every mockk-based Compose test's `@Before setup()` must add a matching `every { viewModel.<prop> } returns MutableStateFlow(<default>)` stub — otherwise the screen crashes at composition because `collectAsStateWithLifecycle()` runs on a null property.
- [0] The `:testing` module (and its `pm.bam.gamedeals.testing.fixtures.*` helpers) is wired into `commonTest` only via `KotlinMultiplatformFeatureConventionPlugin`; `androidInstrumentedTest` does NOT get it. Instrumented tests must either construct domain models directly (with a local helper if needed) or mockk them — don't import fixtures.

- [0] In commonMain previews use `import org.jetbrains.compose.ui.tooling.preview.Preview` with `@file:Suppress("DEPRECATION")`. - [0] CMP `@Preview` `expect` annotation lacks `uiMode` / `device` / `showSystemUi` parameters on non-Android targets — meta-annotations carrying those (e.g. `@DarkAndLightPreview`, `@PhonePortrait`) cannot be plain `annotation class` declarations in commonMain. They require `expect`/`actual`: commonMain `expect annotation class X()`, androidMain `actual` stacking `@Preview(uiMode=…, device=…)` using `androidx.compose.ui.tooling.preview.Preview`, iosMain `actual` as a no-op.
- [0] `androidx.compose.*` packages (foundation, ui, material3, runtime) are cross-platform KMP artifacts in this project via JetBrains' `compose.*` re-publishing — `androidx.compose.foundation.isSystemInDarkTheme()` and similar work fine in commonMain on iOS. The `androidx.*` prefix is misleading; don't assume it means Android-only.
- [0] Cross-module Compose stability inference does NOT work in K2 / Kotlin 2.2.21 — `data class` types in `:domain` are seen as `Stability.Unstable` from every consumer feature module regardless of how structurally stable they are. Required outcome: `@Immutable` on every consumed model (paired with `implementation(libs.compose.runtime)` on `:domain` commonMain). `stability_config.conf` is strictly weaker — only `@Immutable` enables lambda memoization + static expression detection.
- [0] Compose Compiler reports/metrics live in `build-logic/.../ComposeCompilerReports.kt` and are gated behind `-Pgamedeals.composeReports=true` to keep day-to-day builds fast. Configured via `extensions.configure<org.jetbrains.kotlin.compose.compiler.gradle.ComposeCompilerGradlePluginExtension> { reportsDestination/metricsDestination }`. Output lands under `<module>/build/compose-reports/` and `compose-metrics/`.
- [0] When verifying Compose stability after a change, an upstream `:domain` annotation-only edit does NOT invalidate downstream feature modules' compose-reports — Gradle's incremental compile sees them as UP-TO-DATE. Force `--rerun-tasks` (or delete the per-module `build/compose-reports/` dirs) before reading the post-change reports, otherwise you'll see stale "unstable" verdicts on already-fixed types.

Avoid file-path globs like `compose-reports/*-classes.txt` inside KDoc; describe them in plain prose instead.

- [0] Explore-agent reports about file locations and version strings can reflect a *feature branch's* state, not `dev`. - [0] `compose-stability-analyzer` Gradle plugin version is hard-coupled to the project's Kotlin version because the plugin publishes per-target KLIBs (incl. iosArm64) whose ABI matches whatever compiler built them. PR #167 tried 0.7.1 against our then-Kotlin 2.2.21 and failed on iOS — 0.7.1's iOS klibs were built with Kotlin 2.3.20. With the Kotlin 2.3.21 bump (PR #169) the project now matches 0.7.5 (built against Kotlin 2.3.21). When bumping Kotlin in the future, check the analyzer upstream's own `libs.versions.toml` and align.
- [0] `compose-stability-analyzer` plugin is wired via convention plugins (`AndroidApplicationConventionPlugin` + `KotlinMultiplatformLibraryComposeConventionPlugin`), so every Compose-using module gets `<variant>StabilityDump` / `<variant>StabilityCheck` automatically. Baseline `.stability` files live at `<module>/stability/<module>-<variant>.stability` and ARE committed to git — they're the regression contract that CI's `stabilityCheck` compares against. Don't gitignore them. After an intentional stability change, run `./gradlew debugStabilityDump` and commit the updated files alongside the source change.
- [0] Project-wide Kotlin compiler flags need to land in THREE places, not one: `build-logic/.../KotlinMultiplatformLibraryConventionPlugin.kt` (all KMP libraries + feature/compose/ksp variants which compose under it), `build-logic/.../AndroidApplicationConventionPlugin.kt` (the `:app` Android-app module, NOT covered by the KMP convention), and `iosApp/build.gradle.kts` (standalone KMP module producing the iOS `ComposeApp` framework — bypasses both convention plugins and configures `compilerOptions { }` inline).
- [0] Kotlin 2.3 `-Xreturn-value-checker=full` is still warning-level, not error-level, despite the name. `=check` is the conservative subset (catches discarded values in obvious shapes); `=full` widens the scope to project-owned function returns. kotlinx.coroutines 1.10.x and stdlib already annotate `launch`/`tryEmit`/`MutableList.add` with `@IgnorableReturnValue`, so those never fire at any level — pre-scans that predict noise from those patterns will over-estimate.
- [0] `fun foo() = viewModelScope.launch { ... }` (single-expression body) infers a `Job` return type that silently coerces to `Unit` at every `() -> Unit` Compose callback call site. The return-value checker flags this. Prefer block body or explicit `: Unit` so the intent (fire-and-forget) is written down and not just inferred.
- [2] KMP feature module tests use `:feature:<name>:testAndroid` for unit (the bare `:test` is ambiguous — candidates are `testAndroid` / `testAndroidHostTest`, NOT `testDebugUnitTest` which is the `:app`/AGP `com.android.application` task name) and `:feature:<name>:connectedAndroidDeviceTest` for device instrumentation (NOT `:connectedDebugAndroidTest`, which is the Android-app-style name). Same task naming applies to `:remote:<vendor>` modules — they use the same `gamedeals.kmp.library` convention plugin.
- [0] Compose Multiplatform Resources generates each `Res.string.<name>` as an extension property on `Res.string` (per source-set, e.g. `String0.commonMain.kt`). - [1] Wide-layout tests using `onDevice().setScreenOrientation(LANDSCAPE)` (e.g. `HomeScreenTest.storeDataLoadedWide`, `GameScreenTest.gameDetailsLoadedWide`) are pre-existing failures on the local Pixel 9 API 37 emulator — the Espresso Device gRPC service isn't enabled. Treat exactly one orientation-test failure per touched module as environmental, not a regression; verify the failure message is `DeviceControllerOperationException: Failed to set screen orientation`.
- [1] `:feature:webview` device tests load `https://example.com` against the real Android `WebView` on the emulator, so the `onPageFinished` (or `onReceivedError`) callback races every assertion that checks "spinner is displayed". The other tests work around it by calling `client.onPageStarted(view, url, null)` inside `invokeOnWebViewClient` to pin `loading = true` BEFORE asserting; any new assertion on the spinner must do the same. Adding ANY semantics modifier inside the WebView's `TopAppBar` tree — `Modifier.semantics { contentDescription = … }` on the `CircularProgressIndicator` OR `Modifier.semantics { heading() }` on the title `Text` — is enough extra recomposition to make the race manifest 1–2 failures per run, so changes to the WebView toolbar's semantics need paired test stabilization or they will flake.
- [0] Kover 0.9.1 (and earlier) is incompatible with AGP 9's `com.android.kotlin.multiplatform.library` plugin — it fails at configuration with `Kover error: Kover requires extension with name 'android'` because the new KMP-library plugin doesn't create a top-level `android` extension. Kover 0.9.2 added KMP-library support, and 0.9.8 has further fixes for project-isolation cases. Pin to ≥ 0.9.8 in this repo.
- [0] Kover filters set in subproject convention plugins do NOT propagate to the root aggregated `koverHtmlReport`. The root project's own `kover { reports { filters { excludes { ... } } } }` block must duplicate the same filter set (in `build.gradle.kts`), otherwise excluded packages (DI, generated resources, etc.) still show up in the aggregate even though per-module reports look correct. Keep both copies in sync — the comment on each block points at the other.
- [0] In Kover filters, prefer `packages("*.di")` over `classes("*.di.*")` for blanket package exclusions. The `classes()` wildcard `*.di.*` did NOT match `pm.bam.gamedeals.common.di.CommonModuleKt` in 0.9.8 (the `Kt` synthetic classes for top-level Koin module functions still appeared in the report); `packages("*.di")` excludes the whole package cleanly.
- [0] `flatMapLatest` / `transformLatest` cancellation does NOT reduce upstream invocation count when the upstream is a mock that returns synchronously. In `flatMapLatestDelayAtLeast(1000) { gamesRepository.searchGames(it) }`, three rapid upstream emissions trigger three `searchGames` calls on Mokkery — cancellation only kicks in at the `pad.join()` suspension, after the eager `async { transformFunction(value) }` has already invoked the mock. Tests that want to verify "rapid calls collapse" must either (a) make the mock truly suspend (e.g. `calls { gate.await() }`) so cancellation can intercept, or (b) assert on the observable side-effect (only the final result reaches downstream) rather than on the mock's call count.
- [0] Enabling AGP coverage for Android instrumented tests under AGP 9's `com.android.kotlin.multiplatform.library` plugin uses the property `enableCoverage: Boolean` on `KotlinMultiplatformAndroidDeviceTest` (the type exposed by `withDeviceTestBuilder { ... }.configure { }`). NOT `enableAndroidTestCoverage` (that's the `com.android.application` plugin's property in the `buildTypes.debug { }` block) and NOT `testCoverageEnabled` (that was AGP-7 era). The two property names are NOT interchangeable — using the wrong one for the wrong plugin compiles but does nothing.
- [0] **Kover 0.9.x does NOT consume Android instrumented-test `.ec` files.** Tracked at Kover issue [Kotlin/kotlinx-kover#96](https://github.com/Kotlin/kotlinx-kover/issues/96) — OPEN, the maintainer's comment on #592 (May 2024) was "running tests on the device is not supported yet. Work is in progress now." Setting `enableCoverage = true` (KMP-library) or `enableAndroidTestCoverage = true` (`:app`) makes AGP write `coverage.ec` files under `<module>/build/outputs/code_coverage/...`, but Kover's `koverGenerateArtifactAndroid` only collects host-test `.ic` files into `<module>/build/kover/android.artifact` and never adds the device `.ec` files. This project's workaround: a parallel `jacocoAndroidTestReport` task at the root that aggregates the `.ec` files into `build/reports/jacoco/androidTest/`. Two reports total — Kover for unit/host tests, JaCoCo for device tests. The Kover/JaCoCo filter sets are kept aligned manually (Kover uses class-FQN wildcards, JaCoCo uses Ant-style path globs).
- [0] Under AGP 9 the ORIGINAL (non-instrumented) `.class` files needed by a `JacocoReport`'s `classDirectories` live at counter-intuitive paths. For `com.android.application` (`:app`) the originals are at `app/build/intermediates/built_in_kotlinc/debug/compileDebugKotlin/classes/` (Kotlin) + `app/build/intermediates/javac/debug/compileDebugJavaWithJavac/classes/` (Java); the JaCoCo-OFFLINE-INSTRUMENTED copies are at `app/build/intermediates/classes/debug/jacocoDebug/dirs/` — the `jacocoDebug` name is misleading, those are *instrumented* (they contain `$jacocoInit`/`$jacocoData` markers), not "for JaCoCo report use." For `com.android.kotlin.multiplatform.library` the originals are at `<module>/build/classes/kotlin/android/main/` and the instrumented copies sit under `<module>/build/.transforms/<hash>/transformed/instrumented_classes/`. Pointing JaCoCo's report task at the instrumented paths produces `IllegalStateException: Cannot process instrumented class … Please supply original non-instrumented classes` — verify a candidate path with `grep -ao jacocoInit <.class file>` before wiring it in.

- [0] Applying a precompiled convention plugin from the `build-logic` included build to the ROOT project (not just to subprojects) drags every `implementation` dep of `build-logic/convention/build.gradle.kts` onto the build's plugin classpath, where it's then visible to ALL subprojects' plugin DSL. Concretely: `build-logic` declares `implementation(libs.compose.multiplatform.gradle.plugin)` (needed at runtime by `KotlinMultiplatformLibraryComposeConventionPlugin` to call `ComposePlugin.Dependencies(target)`). Once root applies any build-logic plugin, `iosApp/build.gradle.kts` blowing up with `Error resolving plugin [id: 'org.jetbrains.compose', version: '1.11.0'] > … plugin is already on the classpath with an unknown version` is the failure mode — its `alias(libs.plugins.compose.multiplatform)` carries a `version.ref` but the plugin is already loaded transitively. - [0] Mokkery 3.x `answering.throws` (e.g. `everySuspend { … } throws Exception()`) prints the thrown stack trace via K/N's stdout. Gradle's K/N test reporter parses stdout via the TeamCity protocol, and the numbered `at <N>  test.kexe ...` frame lines blow past an array bound in the parser — task fails with `Index N out of bounds for length 2`. Workaround: route the throw through the call site via `calls { throw … }` for suspend/non-suspend functions, or `returns flow { throw … }` for Flow-returning mocks. Behavior is equivalent (production catches the throw the same way) but the Mokkery answer-mode no longer prints to stdout. Affects ALL throws-mode usages — `:feature:home`, `:feature:search`, `:feature:store` still have unconverted ones noted as follow-up.
- [0] Worktrees under `.claude/worktrees/<id>/` do NOT inherit the main repo's `local.properties` — Gradle Android Plugin fails at configuration with `SDK location not found. Define a valid SDK location with an ANDROID_HOME environment variable …`. Easiest fix without polluting the worktree: invoke gradle with `ANDROID_HOME=/Users/bam/Library/Android/sdk JAVA_HOME=/Applications/Android\ Studio.app/Contents/jbr/Contents/Home ./gradlew …`. JBR 21 is required even for non-`:app` tasks because the precompiled convention-plugin classpath demands JVM 21.
- [0] Ktor `install(Auth) { bearer { refreshTokens { … } } }` intercepts a 401 response and runs refresh+retry BEFORE `expectSuccess = true` surfaces it as `ResponseException` — so the Auth plugin and `expectSuccess` coexist on the same HttpClient with no extra config. A 401 that survives refresh (refresh threw, or the retried request also 401s) propagates through `expectSuccess` normally and the source-impl try/catch wraps it into `ApiResponse.exception(t)`.
- [0] Ktor `setBody("plain text") { contentType(ContentType.Text.Plain) }` is NOT touched by `ContentNegotiation { json(json) }` — JSON content-negotiation only activates when the runtime body type matches a registered content type, so a raw `String` with `Text.Plain` passes through to the wire untouched. Verified for IGDB Apicalypse queries.
- [0] `OutgoingContent.contentType` on a `String` body sent via `setBody(text) { contentType(ContentType.Text.Plain) }` is observed by `MockEngine` as `text/plain; charset=UTF-8` (Ktor appends the default charset). Assertions like `assertEquals(ContentType.Text.Plain, body.contentType)` fail with `expected:<text/plain> but was:<text/plain; charset=UTF-8>`. Compare `.contentType` and `.contentSubtype` parts separately, or use `startsWith`.
- [0] Vendor credentials for `:remote:<vendor>` KMP modules enter via a Koin-bound data class (e.g. `IgdbCredentials`) populated at the composition root — Android `:app` reads from `BuildConfig.*` fields (sourced from `local.properties` with `System.getenv(...)` CI fallback in `app/build.gradle.kts`) and iOS `iosApp/src/iosMain/.../MainViewController.kt` reads from `NSBundle.mainBundle.objectForInfoDictionaryKey(...)` (sourced from `iosApp/Configuration/Secrets.xcconfig` → Info.plist `$(VAR)` substitution at Xcode build time). Mirrors the `RemoteBuildType` / `remoteModule(currentRemoteBuildType())` twin-bootstrap shape. Do NOT try to read `BuildConfig` from inside a `:remote:*` module — `commonMain` can't see it; it's Android-only.
- [0] kotlinx.serialization treats a `@Serializable` property as **required** based on whether it has a default value, NOT whether its type is nullable. `val summary: String?` (no default) throws `MissingFieldException` if the JSON omits `summary`; `val summary: String? = null` makes the field omittable. When adding a new optional field to an existing DTO that has fixture/mock JSON in tests without the field, supply the default — otherwise every old fixture starts failing deserialization.
- [0] IGDB v4 `/v4/external_games` migrated the store-identifier field from the legacy `category` enum (1=Steam, 5=GOG, etc.) to a separate reference table `/v4/external_game_sources`. The new field name on each external_game record is **`external_game_source`** (an `id` referencing that table — Steam is still id=1). The old `category` field still EXISTS on the schema and is still queryable in Apicalypse, but it is EMPTY on modern records (verified Halo Infinite Steam id 1240440 and Counter-Strike 2 Steam id 730 both have only `external_game_source: 1`, no `category`). A query like `where uid = "$steamId" & category = 1` silently returns `[]` with no error — IGDB returns 200 OK because the query is syntactically valid, but the filter never matches. Use `external_game_source = 1` for Steam lookups going forward. Third-party docs / wrappers documenting `category = 1` for Steam are pre-migration and out of date.
- [0] IGDB v4 `/v4/websites` ran the same `category` → reference-table migration as external_games. The legacy `websites.category` int enum is empty on modern records (verified Halo Infinite returns `category: null` on all 13 website rows). The new field is **`websites.type`**, a reference to `/v4/website_types`. The integer IDs are largely stable across the migration (1=Official, 3=Wikipedia, 5=Twitter, 6=Twitch, 8=Instagram, 9=YouTube, 13=Steam, 14=Reddit/Subreddit, 18=Discord) with new additions (19=Bluesky, 22=Xbox) and minor `type` string renames ("Official Website", "Subreddit", "Community Wiki"). Query both `websites.type.id` and `websites.type.type` via Apicalypse dot-expansion; map by `type.id` (stable int) rather than the human-readable string. Pattern: when IGDB has a `<table>.category` int enum, assume a `<table>.type` reference-table migration may already have happened and probe live before relying on the legacy field.
- [0] Cross-feature URL utilities (host parsing, brand-keys, favicon URL resolution) live in **`:common/.../favicon`** as a `FaviconResolver` interface + `FaviconResolverImpl`, bound via `commonModule` and injected into consumer ViewModels through Koin. - [0] Before adding a Compose UI library to a `:feature:*` module, verify it ships `iosArm64` + `iosSimulatorArm64` KLIBs — many popular Android Compose libraries are still Android-only. Verified May 2026: `me.saket.telephoto` (the de-facto Android pinch-to-zoom library) is Android-only — KMP support issue [#9](https://github.com/saket/telephoto/issues/9) has been open for >1 year, the iOS PR [#102](https://github.com/saket/telephoto/pull/102) was abandoned in late 2024.
