- [0] Compose `Row.verticalAlignment = CenterVertically` centers within the inner content area, not the padded outer bounds ‚Äî for inter-row spacing on a centered row, use `padding(vertical = X)` not `padding(bottom = X)`, or the row's content looks pushed toward the top.
- [0] Compose `onNodeWithText` is exact-match by default ‚Äî when a single `Text` uses `buildAnnotatedString` to concatenate spans, the whole concatenated string is one node; assertions must use the full string or pass `substring = true`.
- [0] When a Composable gets a new required parameter (e.g. `onViewFavourites`), every test invocation of that Composable must add the param OR a default value must be supplied ‚Äî Compose instrumented tests fail at compile, not at test discovery, so a stale test file silently rots a whole module's Android tests.
- [0] When a ViewModel adds a new collected StateFlow property, every mockk-based Compose test's `@Before setup()` must add a matching `every { viewModel.<prop> } returns MutableStateFlow(<default>)` stub ‚Äî otherwise the screen crashes at composition because `collectAsStateWithLifecycle()` runs on a null property.
- [0] The `:testing` module (and its `pm.bam.gamedeals.testing.fixtures.*` helpers) is wired into `commonTest` only via `KotlinMultiplatformFeatureConventionPlugin`; `androidInstrumentedTest` does NOT get it. Instrumented tests must either construct domain models directly (with a local helper if needed) or mockk them ‚Äî don't import fixtures.
- [0] `common/ui/src/commonMain/.../PreviewData.kt` is the shared preview-fixture registry (`PreviewStore`/`PreviewDeal`/`PreviewGiveaway`/`PreviewRelease`/`PreviewFavourite`/`PreviewDealDetails`/`PreviewGameDetails`/`PreviewGame`) ‚Äî reuse these in commonMain previews instead of defining module-local `previewSample*` constants. Add new entries here when introducing a new domain type.
- [0] In commonMain previews use `import org.jetbrains.compose.ui.tooling.preview.Preview` with `@file:Suppress("DEPRECATION")`. The CMP-1.10 deprecation warning that recommends `androidx.compose.ui.tooling.preview.Preview` is correct only on androidMain ‚Äî the AndroidX-qualified import does not resolve on the iOS commonMain compile.
- [0] CMP `@Preview` `expect` annotation lacks `uiMode` / `device` / `showSystemUi` parameters on non-Android targets ‚Äî meta-annotations carrying those (e.g. `@DarkAndLightPreview`, `@PhonePortrait`) cannot be plain `annotation class` declarations in commonMain. They require `expect`/`actual`: commonMain `expect annotation class X()`, androidMain `actual` stacking `@Preview(uiMode=‚Ä¶, device=‚Ä¶)` using `androidx.compose.ui.tooling.preview.Preview`, iosMain `actual` as a no-op.
- [0] `androidx.compose.*` packages (foundation, ui, material3, runtime) are cross-platform KMP artifacts in this project via JetBrains' `compose.*` re-publishing ‚Äî `androidx.compose.foundation.isSystemInDarkTheme()` and similar work fine in commonMain on iOS. The `androidx.*` prefix is misleading; don't assume it means Android-only.
- [0] Cross-module Compose stability inference does NOT work in K2 / Kotlin 2.2.21 ‚Äî `data class` types in `:domain` are seen as `Stability.Unstable` from every consumer feature module regardless of how structurally stable they are. Required outcome: `@Immutable` on every consumed model (paired with `implementation(libs.compose.runtime)` on `:domain` commonMain). `stability_config.conf` is strictly weaker ‚Äî only `@Immutable` enables lambda memoization + static expression detection.
- [0] Compose Compiler reports/metrics live in `build-logic/.../ComposeCompilerReports.kt` and are gated behind `-Pgamedeals.composeReports=true` to keep day-to-day builds fast. Configured via `extensions.configure<org.jetbrains.kotlin.compose.compiler.gradle.ComposeCompilerGradlePluginExtension> { reportsDestination/metricsDestination }`. Output lands under `<module>/build/compose-reports/` and `compose-metrics/`.
- [0] When verifying Compose stability after a change, an upstream `:domain` annotation-only edit does NOT invalidate downstream feature modules' compose-reports ‚Äî Gradle's incremental compile sees them as UP-TO-DATE. Force `--rerun-tasks` (or delete the per-module `build/compose-reports/` dirs) before reading the post-change reports, otherwise you'll see stale "unstable" verdicts on already-fixed types.
- [0] Writing `/*` (slash-star) inside a KDoc block opens a nested block-comment that swallows the closing `*/`, leaving the outer KDoc unclosed (compile error "Syntax error: Unclosed comment"). Avoid file-path globs like `compose-reports/*-classes.txt` inside KDoc; describe them in plain prose instead.
- [0] Room `@ProvidedTypeConverter` works seamlessly with `kotlinx.collections.immutable.ImmutableList<T>` as parameter/return type ‚Äî the `GiveawayPlatformsConverter` swap from `List<GiveawayPlatform>` ‚Üí `ImmutableList<GiveawayPlatform>` compiles cleanly under KSP on both Android and iOS, no extra adapter wiring needed.
- [0] Explore-agent reports about file locations and version strings can reflect a *feature branch's* state, not `dev`. Cross-check any cited path/line/version against `git ls-tree dev -- <path>` (or `git show dev:<path>`) before trusting it ‚Äî e.g. an agent on this repo confidently cited `compose-stability-analyzer = "0.7.1"` at `libs.versions.toml:117` and a `StabilityAnalyzerCompat.kt` file in `build-logic`, both of which existed only on the unmerged `chore/add-compose-stability-analyzer` branch.
- [0] `compose-stability-analyzer` Gradle plugin version is hard-coupled to the project's Kotlin version because the plugin publishes per-target KLIBs (incl. iosArm64) whose ABI matches whatever compiler built them. PR #167 tried 0.7.1 against our then-Kotlin 2.2.21 and failed on iOS ‚Äî 0.7.1's iOS klibs were built with Kotlin 2.3.20. With the Kotlin 2.3.21 bump (PR #169) the project now matches 0.7.5 (built against Kotlin 2.3.21). When bumping Kotlin in the future, check the analyzer upstream's own `libs.versions.toml` and align.
- [0] `compose-stability-analyzer` plugin is wired via convention plugins (`AndroidApplicationConventionPlugin` + `KotlinMultiplatformLibraryComposeConventionPlugin`), so every Compose-using module gets `<variant>StabilityDump` / `<variant>StabilityCheck` automatically. Baseline `.stability` files live at `<module>/stability/<module>-<variant>.stability` and ARE committed to git ‚Äî they're the regression contract that CI's `stabilityCheck` compares against. Don't gitignore them. After an intentional stability change, run `./gradlew debugStabilityDump` and commit the updated files alongside the source change.
- [0] Project-wide Kotlin compiler flags need to land in THREE places, not one: `build-logic/.../KotlinMultiplatformLibraryConventionPlugin.kt` (all KMP libraries + feature/compose/ksp variants which compose under it), `build-logic/.../AndroidApplicationConventionPlugin.kt` (the `:app` Android-app module, NOT covered by the KMP convention), and `iosApp/build.gradle.kts` (standalone KMP module producing the iOS `ComposeApp` framework ‚Äî bypasses both convention plugins and configures `compilerOptions { }` inline).
- [0] Kotlin 2.3 `-Xreturn-value-checker=full` is still warning-level, not error-level, despite the name. `=check` is the conservative subset (catches discarded values in obvious shapes); `=full` widens the scope to project-owned function returns. kotlinx.coroutines 1.10.x and stdlib already annotate `launch`/`tryEmit`/`MutableList.add` with `@IgnorableReturnValue`, so those never fire at any level ‚Äî pre-scans that predict noise from those patterns will over-estimate.
- [0] `fun foo() = viewModelScope.launch { ... }` (single-expression body) infers a `Job` return type that silently coerces to `Unit` at every `() -> Unit` Compose callback call site. The return-value checker flags this. Prefer block body or explicit `: Unit` so the intent (fire-and-forget) is written down and not just inferred.
- [0] KMP feature module tests use `:feature:<name>:testAndroid` for unit (the bare `:test` is ambiguous ‚Äî candidates are `testAndroid` / `testAndroidHostTest`) and `:feature:<name>:connectedAndroidDeviceTest` for device instrumentation (NOT `:connectedDebugAndroidTest`, which is the Android-app-style name).
- [0] Compose Multiplatform Resources generates each `Res.string.<name>` as an extension property on `Res.string` (per source-set, e.g. `String0.commonMain.kt`). Each must be imported individually ‚Äî e.g. `import pm.bam.gamedeals.feature.home.generated.resources.home_screen_X`. Adding a new string to `strings.xml` without adding the import produces an "Unresolved reference" error even though the generated file already contains the property.
- [1] Wide-layout tests using `onDevice().setScreenOrientation(LANDSCAPE)` (e.g. `HomeScreenTest.storeDataLoadedWide`, `GameScreenTest.gameDetailsLoadedWide`) are pre-existing failures on the local Pixel 9 API 37 emulator ‚Äî the Espresso Device gRPC service isn't enabled. Treat exactly one orientation-test failure per touched module as environmental, not a regression; verify the failure message is `DeviceControllerOperationException: Failed to set screen orientation`.
- [1] `:feature:webview` device tests load `https://example.com` against the real Android `WebView` on the emulator, so the `onPageFinished` (or `onReceivedError`) callback races every assertion that checks "spinner is displayed". The other tests work around it by calling `client.onPageStarted(view, url, null)` inside `invokeOnWebViewClient` to pin `loading = true` BEFORE asserting; any new assertion on the spinner must do the same. Adding ANY semantics modifier inside the WebView's `TopAppBar` tree ‚Äî `Modifier.semantics { contentDescription = ‚Ä¶ }` on the `CircularProgressIndicator` OR `Modifier.semantics { heading() }` on the title `Text` ‚Äî is enough extra recomposition to make the race manifest 1‚Äì2 failures per run, so changes to the WebView toolbar's semantics need paired test stabilization or they will flake.
- [0] Kover 0.9.1 (and earlier) is incompatible with AGP 9's `com.android.kotlin.multiplatform.library` plugin ‚Äî it fails at configuration with `Kover error: Kover requires extension with name 'android'` because the new KMP-library plugin doesn't create a top-level `android` extension. Kover 0.9.2 added KMP-library support, and 0.9.8 has further fixes for project-isolation cases. Pin to ‚â• 0.9.8 in this repo.
- [0] Kover filters set in subproject convention plugins do NOT propagate to the root aggregated `koverHtmlReport`. The root project's own `kover { reports { filters { excludes { ... } } } }` block must duplicate the same filter set (in `build.gradle.kts`), otherwise excluded packages (DI, generated resources, etc.) still show up in the aggregate even though per-module reports look correct. Keep both copies in sync ‚Äî the comment on each block points at the other.
- [0] In Kover filters, prefer `packages("*.di")` over `classes("*.di.*")` for blanket package exclusions. The `classes()` wildcard `*.di.*` did NOT match `pm.bam.gamedeals.common.di.CommonModuleKt` in 0.9.8 (the `Kt` synthetic classes for top-level Koin module functions still appeared in the report); `packages("*.di")` excludes the whole package cleanly.
- [0] `flatMapLatest` / `transformLatest` cancellation does NOT reduce upstream invocation count when the upstream is a mock that returns synchronously. In `flatMapLatestDelayAtLeast(1000) { gamesRepository.searchGames(it) }`, three rapid upstream emissions trigger three `searchGames` calls on Mokkery ‚Äî cancellation only kicks in at the `pad.join()` suspension, after the eager `async { transformFunction(value) }` has already invoked the mock. Tests that want to verify "rapid calls collapse" must either (a) make the mock truly suspend (e.g. `calls { gate.await() }`) so cancellation can intercept, or (b) assert on the observable side-effect (only the final result reaches downstream) rather than on the mock's call count.
- [0] Enabling AGP coverage for Android instrumented tests under AGP 9's `com.android.kotlin.multiplatform.library` plugin uses the property `enableCoverage: Boolean` on `KotlinMultiplatformAndroidDeviceTest` (the type exposed by `withDeviceTestBuilder { ... }.configure { }`). NOT `enableAndroidTestCoverage` (that's the `com.android.application` plugin's property in the `buildTypes.debug { }` block) and NOT `testCoverageEnabled` (that was AGP-7 era). The two property names are NOT interchangeable ‚Äî using the wrong one for the wrong plugin compiles but does nothing.
- [0] **Kover 0.9.x does NOT consume Android instrumented-test `.ec` files.** Tracked at Kover issue [Kotlin/kotlinx-kover#96](https://github.com/Kotlin/kotlinx-kover/issues/96) ‚Äî OPEN, the maintainer's comment on #592 (May 2024) was "running tests on the device is not supported yet. Work is in progress now." Setting `enableCoverage = true` (KMP-library) or `enableAndroidTestCoverage = true` (`:app`) makes AGP write `coverage.ec` files under `<module>/build/outputs/code_coverage/...`, but Kover's `koverGenerateArtifactAndroid` only collects host-test `.ic` files into `<module>/build/kover/android.artifact` and never adds the device `.ec` files. **Until Kover #96 ships, the only paths to UI-test coverage are (a) wire JaCoCo directly as a parallel tool, or (b) move UI tests to JVM (Robolectric / Roborazzi / Paparazzi) so they ride the existing Kover pipeline.**
