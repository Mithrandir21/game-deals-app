- [0] Compose `Row.verticalAlignment = CenterVertically` centers within the inner content area, not the padded outer bounds — for inter-row spacing on a centered row, use `padding(vertical = X)` not `padding(bottom = X)`, or the row's content looks pushed toward the top.
- [0] Compose `onNodeWithText` is exact-match by default — when a single `Text` uses `buildAnnotatedString` to concatenate spans, the whole concatenated string is one node; assertions must use the full string or pass `substring = true`.
- [0] When a Composable gets a new required parameter (e.g. `onViewFavourites`), every test invocation of that Composable must add the param OR a default value must be supplied — Compose instrumented tests fail at compile, not at test discovery, so a stale test file silently rots a whole module's Android tests.
- [0] When a ViewModel adds a new collected StateFlow property, every mockk-based Compose test's `@Before setup()` must add a matching `every { viewModel.<prop> } returns MutableStateFlow(<default>)` stub — otherwise the screen crashes at composition because `collectAsStateWithLifecycle()` runs on a null property.
- [0] The `:testing` module (and its `pm.bam.gamedeals.testing.fixtures.*` helpers) is wired into `commonTest` only via `KotlinMultiplatformFeatureConventionPlugin`; `androidInstrumentedTest` does NOT get it. Instrumented tests must either construct domain models directly (with a local helper if needed) or mockk them — don't import fixtures.
- [0] `common/ui/src/commonMain/.../PreviewData.kt` is the shared preview-fixture registry (`PreviewStore`/`PreviewDeal`/`PreviewGiveaway`/`PreviewRelease`/`PreviewFavourite`/`PreviewDealDetails`/`PreviewGameDetails`/`PreviewGame`) — reuse these in commonMain previews instead of defining module-local `previewSample*` constants. Add new entries here when introducing a new domain type.
- [0] In commonMain previews use `import org.jetbrains.compose.ui.tooling.preview.Preview` with `@file:Suppress("DEPRECATION")`. The CMP-1.10 deprecation warning that recommends `androidx.compose.ui.tooling.preview.Preview` is correct only on androidMain — the AndroidX-qualified import does not resolve on the iOS commonMain compile.
- [0] CMP `@Preview` `expect` annotation lacks `uiMode` / `device` / `showSystemUi` parameters on non-Android targets — meta-annotations carrying those (e.g. `@DarkAndLightPreview`, `@PhonePortrait`) cannot be plain `annotation class` declarations in commonMain. They require `expect`/`actual`: commonMain `expect annotation class X()`, androidMain `actual` stacking `@Preview(uiMode=…, device=…)` using `androidx.compose.ui.tooling.preview.Preview`, iosMain `actual` as a no-op.
- [0] `androidx.compose.*` packages (foundation, ui, material3, runtime) are cross-platform KMP artifacts in this project via JetBrains' `compose.*` re-publishing — `androidx.compose.foundation.isSystemInDarkTheme()` and similar work fine in commonMain on iOS. The `androidx.*` prefix is misleading; don't assume it means Android-only.
- [0] Cross-module Compose stability inference does NOT work in K2 / Kotlin 2.2.21 — `data class` types in `:domain` are seen as `Stability.Unstable` from every consumer feature module regardless of how structurally stable they are. Required outcome: `@Immutable` on every consumed model (paired with `implementation(libs.compose.runtime)` on `:domain` commonMain). `stability_config.conf` is strictly weaker — only `@Immutable` enables lambda memoization + static expression detection.
- [0] Compose Compiler reports/metrics live in `build-logic/.../ComposeCompilerReports.kt` and are gated behind `-Pgamedeals.composeReports=true` to keep day-to-day builds fast. Configured via `extensions.configure<org.jetbrains.kotlin.compose.compiler.gradle.ComposeCompilerGradlePluginExtension> { reportsDestination/metricsDestination }`. Output lands under `<module>/build/compose-reports/` and `compose-metrics/`.
- [0] When verifying Compose stability after a change, an upstream `:domain` annotation-only edit does NOT invalidate downstream feature modules' compose-reports — Gradle's incremental compile sees them as UP-TO-DATE. Force `--rerun-tasks` (or delete the per-module `build/compose-reports/` dirs) before reading the post-change reports, otherwise you'll see stale "unstable" verdicts on already-fixed types.
- [0] Writing `/*` (slash-star) inside a KDoc block opens a nested block-comment that swallows the closing `*/`, leaving the outer KDoc unclosed (compile error "Syntax error: Unclosed comment"). Avoid file-path globs like `compose-reports/*-classes.txt` inside KDoc; describe them in plain prose instead.
- [0] Room `@ProvidedTypeConverter` works seamlessly with `kotlinx.collections.immutable.ImmutableList<T>` as parameter/return type — the `GiveawayPlatformsConverter` swap from `List<GiveawayPlatform>` → `ImmutableList<GiveawayPlatform>` compiles cleanly under KSP on both Android and iOS, no extra adapter wiring needed.
- [0] Explore-agent reports about file locations and version strings can reflect a *feature branch's* state, not `dev`. Cross-check any cited path/line/version against `git ls-tree dev -- <path>` (or `git show dev:<path>`) before trusting it — e.g. an agent on this repo confidently cited `compose-stability-analyzer = "0.7.1"` at `libs.versions.toml:117` and a `StabilityAnalyzerCompat.kt` file in `build-logic`, both of which existed only on the unmerged `chore/add-compose-stability-analyzer` branch.
- [0] `compose-stability-analyzer` Gradle plugin version is hard-coupled to the project's Kotlin version because the plugin publishes per-target KLIBs (incl. iosArm64) whose ABI matches whatever compiler built them. PR #167 tried 0.7.1 against our then-Kotlin 2.2.21 and failed on iOS — 0.7.1's iOS klibs were built with Kotlin 2.3.20. With the Kotlin 2.3.21 bump (PR #169) the project now matches 0.7.5 (built against Kotlin 2.3.21). When bumping Kotlin in the future, check the analyzer upstream's own `libs.versions.toml` and align.
- [0] `compose-stability-analyzer` plugin is wired via convention plugins (`AndroidApplicationConventionPlugin` + `KotlinMultiplatformLibraryComposeConventionPlugin`), so every Compose-using module gets `<variant>StabilityDump` / `<variant>StabilityCheck` automatically. Baseline `.stability` files live at `<module>/stability/<module>-<variant>.stability` and ARE committed to git — they're the regression contract that CI's `stabilityCheck` compares against. Don't gitignore them. After an intentional stability change, run `./gradlew debugStabilityDump` and commit the updated files alongside the source change.
- [0] Project-wide Kotlin compiler flags need to land in THREE places, not one: `build-logic/.../KotlinMultiplatformLibraryConventionPlugin.kt` (all KMP libraries + feature/compose/ksp variants which compose under it), `build-logic/.../AndroidApplicationConventionPlugin.kt` (the `:app` Android-app module, NOT covered by the KMP convention), and `iosApp/build.gradle.kts` (standalone KMP module producing the iOS `ComposeApp` framework — bypasses both convention plugins and configures `compilerOptions { }` inline).
- [0] Kotlin 2.3 `-Xreturn-value-checker=full` is still warning-level, not error-level, despite the name. `=check` is the conservative subset (catches discarded values in obvious shapes); `=full` widens the scope to project-owned function returns. kotlinx.coroutines 1.10.x and stdlib already annotate `launch`/`tryEmit`/`MutableList.add` with `@IgnorableReturnValue`, so those never fire at any level — pre-scans that predict noise from those patterns will over-estimate.
- [0] `fun foo() = viewModelScope.launch { ... }` (single-expression body) infers a `Job` return type that silently coerces to `Unit` at every `() -> Unit` Compose callback call site. The return-value checker flags this. Prefer block body or explicit `: Unit` so the intent (fire-and-forget) is written down and not just inferred.
